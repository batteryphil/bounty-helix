Q9: task_completion_rate has been exactly 0.5 across ALL 69 snapshots for 36
    hours. The variance is zero. This is statistically impossible for a real
    measurement. It appears hardcoded. A constant in the fitness formula
    provides neither signal nor noise — but it consumes a weight coefficient
    that could be given to a real metric. Should it be zeroed out until
    properly instrumented? And what would proper instrumentation look like
    for an agent that sets its own goals?

Q10: The fitness score dropped from ~0.61 to ~0.50 after the RESTING interval
     was reduced from 900s to 60s. This appears to be because:
     (a) avg_response_length dropped (200-token pulses vs 512-token interactive)
     (b) shorter pulses mean less context, which affects other metrics
     Is avg_response_length a meaningful fitness component for an autonomous agent?
     A longer response is not necessarily better reasoning. Should it be removed
     or replaced with something like reasoning_steps_per_response?

Q11: The agent wrote 22 tools in ~36 hours. ZERO experience tuples were
     collected despite those writes. The quality gate requires ≥2/3 criteria:
       - tool_execution_success: never triggered (tool call rate was 0 until 16:12)
       - novel_belief_added: never triggered (belief rate = 0.00)
       - fitness_delta > 0.05: never triggered (all deltas = 0.000)
     This is a complete deadlock: quality tuples require tool calls, but the
     mandate didn't produce calls until pulse 650 (after 36 hours). Now that
     tool calls are beginning, should the gate be temporarily loosened to
     seed the tuple pool? E.g., accept tool_execution_success ALONE (1/3)?
     Risk: collects lower-quality tuples. Benefit: LoRA pipeline gets data.

Q12: The agent's curiosity is increasingly self-referential. 14 of 58 unique
     questions (24%) read its own repo files or the user's other repos.
     This is healthy self-awareness. But in the last curiosity batch, the
     same question repeated 5 times in a row:
       "How do biological neural networks handle memory consolidation during sleep?"
     The knowledge base has 1,307 entries but only 58 unique questions — average
     22.5 entries per unique question. Something is causing repeat research on
     the same topics. Is the deduplication in the curiosity picker failing?
     Or is the semantic similarity check (if any) not working?

Q13: The belief content truncation bug (Section 3.5): stored beliefs begin
     mid-sentence. Example: "plore how I can utilize tools and APIs..."
     The belief was likely "I should explore how I can utilize tools and APIs..."
     The regex is extracting from an offset. Is this losing meaningful belief
     content, or is the full content stored in raw form and only the display
     is truncated? Does this affect the belief detector's novelty check
     (duplicate detection compares content strings — a truncated belief
     would never match its untruncated version, causing duplicates)?

Q14: The SIE has written 392 code write entries and 189 tool additions, but
     all at Δfitness = 0.000. Every single commit. The SIE is productive
     (writes, tests pass, commits) but has zero measurable impact.
     The evaluator measures import-level success, not runtime behavior.
     Is there a way to make the fitness evaluator call the tool it just wrote
     with a test payload and measure the response? E.g., after writing
     system_health.py, the evaluator should call system_health.get_metrics()
     and confirm it returns valid data, then give +0.05 for a real execution.

Q15: The agent wrote behavior_tracker.py, curiosity_tracker.py, fitness_tracker.py

     but does Hermes-3 understand that self-introspection tools are available
